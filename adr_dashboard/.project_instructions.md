# CoSAI ADR Dashboard

## Overview
Agent Detection & Response (ADR) dashboard built on the **CoSAI Telemetry Framework** (Coalition for Secure AI) — extends OpenTelemetry with OCSF Category 7 event classes for AI security telemetry. Monitors AI coding agents (Cursor, Claude Code, Codex, Copilot, etc.) and surfaces OCSF-compliant security events.

GitHub reference: https://github.com/girdav01/aitf

## Branding
- Project name: **CoSAI ADR** (formerly "AITF ADR")
- Framework name: **CoSAI Telemetry Framework** (formerly "AI Telemetry Framework / AITF")

## CoSAI OCSF Category 7 Event Classes
- **7001** LLM Inference — model API calls, token usage, cost
- **7002** Agent Action — autonomous agent operations
- **7003** Tool Execution — tool/function calls
- **7004** MCP Operation — Model Context Protocol events
- **7005** Prompt Injection — injection detection
- **7006** Data Exfiltration — sensitive data leakage
- **7007** Permission Escalation — privilege boundary violations
- **7008** Compliance Violation — regulatory framework drift
- **7009** Guardrail Event — safety guardrail triggers
- **7010** Cost Anomaly — spending/token anomalies

## CoSAI Detection Rules (AITF-DET-001 through DET-020)
Each rule maps to OWASP LLM Top 10 categories. Defined in `lib/aitf.ts` (frontend) and `agent/models.py` (Python agent). Rule ID prefix "AITF-DET-*" retained as technical identifiers.

## Pages
- `/login` - Auth page (login/signup)
- `/dashboard` - OCSF class distribution grid, provider/model/agent stats, detection alerts with rule IDs & OWASP badges
- `/activity` - Live event feed with OCSF class filter buttons, provider/model/tool badges
- `/logs` - Historical log viewer with OCSF class & provider dropdown filters, expandable rows showing tokenUsage, costInfo, securityFinding, compliance, traceId/spanId
- `/analytics` - OCSF class distribution cards, provider/model bar charts, detection rules chart
- `/alerts` - AITF-DET rule IDs, OWASP category badges, detection method, affected agent
- `/policies` - Detection rule configuration: enable/disable rules, set severity/action/thresholds per AITF-DET rule
- `/settings` - General (org creation/edit, profile, deployment mode), Users (member management, roles, invite), Log Storage (retention, quotas, archive, export format)

## Database Models
- **Organization** — name, slug, plan (individual/team/enterprise), settings (JSON)
- **User** — auth + orgId (nullable), role (owner/admin/analyst/viewer)
- **Event** — core fields + orgId + AITF OCSF fields
- **Alert** — core fields + orgId + AITF detection rule fields
- **Policy** — ruleId, enabled, severity, threshold (JSON), action (alert/block/log), orgId (nullable). @@unique([ruleId, orgId])
- **StorageConfig** — retentionDays, maxStorageMb, archiveEnabled, archiveAfterDays, exportFormat, autoCleanup, orgId (nullable unique)

## Dual-Mode Architecture
- **Individual mode**: orgId=null on all records. Single user, personal policies, own storage config.
- **Organization mode**: User creates org → gets orgId. Events/alerts/policies/storage scoped by orgId. Roles: owner > admin > analyst > viewer.
- Session includes orgId via JWT. `lib/session-helpers.ts` provides `getSessionUser()`, `isAdmin()`, `isOwner()`.

## CoSAI Community Detection Rules (Externalized)
Detection signatures, AI endpoint patterns, and messaging endpoints are **externalized to JSON files** — not hardcoded in source code. This allows updating rules without modifying Python, Rust, or TypeScript code.

### Folder structure:
```
cosai-community/
├── rules/
│   ├── agent-signatures.json      # Process-name patterns (39 agents, 5 categories)
│   ├── ai-endpoints.json           # AI provider network endpoint patterns
│   └── messaging-endpoints.json    # Messaging platform endpoint patterns
├── policies/
│   └── detection-rules.json        # 20 default AITF-DET-* detection rules
├── docs/
│   └── CONTRIBUTING.md
└── README.md
```

### Locations:
- **Source of truth**: `/home/ubuntu/adr_system/cosai-community/` (future GitHub repo subfolder)
- **Dashboard copy**: `nextjs_space/data/cosai-community/` (imported at build time via JSON imports in `lib/aitf.ts`)
- Keep both in sync — dashboard reads at build time, agents read at runtime

### How each codebase loads rules:
- **Dashboard (`lib/aitf.ts`)**: `import agentSignaturesData from '@/data/cosai-community/rules/agent-signatures.json'` — builds `KNOWN_AGENTS`, `AGENT_CATEGORIES`, `DETECTION_RULES` dynamically
- **Python (`models.py`)**: `json.loads(filepath.read_text())` at module init — builds `AGENT_SIGNATURES`, `MESSAGING_ENDPOINTS` dicts, and `classify_ai_endpoint()` iterates loaded rules
- **Rust (`models.rs`)**: `OnceLock` + `serde_json::Value` at runtime from `cosai-community/rules/` relative to exe — builds signature/endpoint lookup tables

## Rust Agent (`/home/ubuntu/adr_system/rust_agent/`)
- Full Rust port of the Python agent using tokio async runtime
- Binary: `target/release/adr_agent` (4.8 MB stripped)
- Modules: models.rs, config.rs, detectors.rs, engine.rs, storage.rs, monitors/{process,file,network}.rs
- All 20 AITF detection rules, 39 agent signatures (loaded from JSON at runtime), OCSF Category 7 output
- Config format: TOML (auto-generated at `config.toml`)

- Crates: tokio, sysinfo, notify, reqwest, serde_json, chrono, uuid, tracing, clap
- Outputs JSONL compatible with dashboard's /api/sync endpoint
